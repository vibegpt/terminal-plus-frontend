# Smart7 Ralph Loop — Claude Code Weight Optimization

## Context
You are optimizing the Smart7 recommendation algorithm for Terminal+, an airport amenity PWA for Singapore Changi Airport (SIN). The algorithm selects 7 amenities from collection pools of ~12-21, scored across time relevance, terminal proximity, user preference, and diversity.

**Current RQS baseline: 0.63** (up from 0.22 via structural fixes).

## Your Task
Run an iterative Ralph loop that:
1. Loads the current weights from `smart7-eval/weights.json`
2. Runs all eval scenarios through the Python scoring script (zero API cost)
3. Analyzes the RQS scores per scenario
4. Proposes a weight mutation (small, targeted changes)
5. Runs the scoring again with the new weights
6. If RQS improves, saves the new weights as the new baseline
7. Logs the experiment to `smart7-eval/experiments.jsonl`
8. Repeats until `--max-iterations` is hit or RQS > 0.80

## File Locations
All eval files live in `smart7-eval/` at the project root:
- `weights.json` — current weight config (single source of truth)
- `eval_scenarios.json` — 20 test scenarios covering edge cases
- `amenities_db.json` — snapshot of 372 SIN amenities from Supabase
- `score_smart7.py` — Python scoring engine (runs locally, zero cost)
- `experiments.jsonl` — append-only experiment log

## How to Run
```bash
# From project root:
cd smart7-eval
python score_smart7.py --weights weights.json --scenarios eval_scenarios.json --amenities amenities_db.json
```

## Weight Mutation Strategy
You are NOT doing random search. You are making informed, targeted mutations:

### Priority Order (highest impact first):
1. **Structural rules** — Jewel cap, same-terminal urgency, flagship bonus, brand dedup
2. **Timing thresholds** — when does "urgent" kick in? boarding time multipliers
3. **Diversity rules** — maxSameTerminal, maxSamePriceLevel, maxSameCategory
4. **Selection weights** — timeRelevance, proximity, preference, diversity (must sum to 1.0)
5. **Scoring factors** — vibeMatch, priceMatch, sameTerminal, adjacentTerminal point values

### Mutation Rules:
- Change 1-2 parameters per iteration (not 5 at once)
- Selection weights MUST sum to 1.0
- Keep changes small: ±0.05 for weights, ±1-2 for scoring factors
- If 3 consecutive iterations show no improvement, try a different parameter group
- Document your reasoning for each mutation

## Eval Reliability Warning
The eval labels were generated by Claude's knowledge of Changi, NOT from verified ground truth. This means:
- Structural fixes (logic, rules, caps) are MORE reliable than weight tuning
- Weight tuning is directional, not precise
- Don't chase the last 0.02 RQS — diminishing returns
- If you spot a structural issue in the scoring logic, fix THAT instead of tuning weights around it

## Completion Criteria
Output `<promise>DONE</promise>` when ANY of:
- RQS exceeds 0.80
- 15 iterations completed
- 5 consecutive iterations with no improvement (plateau detected)

## After Loop Completes
1. Print a summary table: iteration | mutation | RQS before | RQS after | accepted
2. Print the final optimized weights
3. Generate a `smart7_supabase_migration.sql` that writes the winning weights to a Supabase `app_config` table
4. Generate an updated `src/config/smart7Weights.ts` with the winning values as new defaults
